# Dev Flow (DndMcpAICsharpFun)

## Overview

The repeatable cycle for changing this project: one ASP.NET Core .NET 10 host serving API + MCP + Blazor UI, with the ingestion/extraction → Postgres/Qdrant pipeline. Each change runs the same loop and **lands directly on `main`** (single-dev, no feature branches).

**Core principle:** No change is "done" on assertion. Done = gates run, output seen, all green. For data/extraction changes, done = **validated on a real book**, not just unit-green. Evidence before claims.

## When to Use

- Starting any feature/change/spec.
- Before writing implementation code.
- Before claiming a change is "done", "fixed", "working", "green", or "validated".
- Before committing or archiving an OpenSpec change.

## The Cycle (never skip, never collapse)

1. **Brainstorm** (`superpowers:brainstorming`) — full dialogue, one question at a time, propose 2-3 approaches, get explicit approval. Design is captured in the OpenSpec change, never loose `docs/`.
2. **Propose** (`opsx:propose`) — create the OpenSpec change in `openspec/changes/<name>/` (proposal, design, specs, tasks).
3. **Plan** (`superpowers:writing-plans`) — detailed bite-sized plan saved to `openspec/changes/<name>/plan.md`.
4. **Implement** (`superpowers:subagent-driven-development`) — fresh implementer + reviewer subagent per task, per-task TDD, fix loop on Critical/Important, final whole-branch review on the most capable model. **All on `main`** — commit each reviewed task straight to main (commit autonomy granted; `mem:workflow/work_on_main`).
5. **Finish** (user says "commit" / "finish X" on a done spec): **commit → `openspec archive <name> -y` → run `skill-optimizer` → refresh the roadmap memory** (`mem:workflow/finishing_a_spec`). **`ingest-entities` is extraction/canonical-only** — run it as the last step when the change adds/edits `books/canonical/*` entities (to project them into `dnd_entities`); retrieval-only, refactor, or endpoint changes SKIP it.

## Serena is MANDATORY for code

All `.cs` reads/edits go through Serena MCP (`find_symbol`, `replace_symbol_body`, `replace_content`, `find_referencing_symbols`). Built-in Read/Edit/Write on code files is **forbidden** (`mem:workflow/serena_first`). Every implementer/reviewer subagent prompt must include the CRITICAL-Serena block + an `initial_instructions` call.

## Quality Gates (must be GREEN before "done")

```bash
dotnet build                                          # 0 warnings (warnings-as-errors via Directory.Build.props)
dotnet test --filter "FullyQualifiedName!~Persistence"  # non-persistence suite (no DB needed)
dotnet test                                           # FULL suite — needs Docker (Testcontainers postgres:18)
dotnet format --verify-no-changes                     # formatting/analyzers
pnpm lint:md                                          # markdown: 0 error(s) (after pnpm lint:md:fix); see /lint-md
```

- Persistence tests need Docker running (Testcontainers + Respawn). Non-persistence suite needs nothing.
- Endpoint change (`MapGet/Post/Put/Delete`) → update `DndMcpAICsharpFun.http` AND `dnd-mcp-api.insomnia.json` in the same commit.
- **SECURITY REVIEW (HTTP / endpoint / config changes) — mandatory.** The `security-guidance` plugin runs pattern warnings on every edit and an async LLM review on `git commit`/`git push`. Do NOT dismiss it. For any HTTP endpoint (`MapGet/Post/Put/Delete`), request-body/query/header handling, admin-key/auth surface, MCP tool surface, or `Config/appsettings.*`/env/`docker-compose*` change, explicitly review the diff for: **authz/admin-gate gaps, injection/SSRF, secret exposure, and unsafe config/CORS/rate-limit**. If the commit/push hook re-wakes with findings, address or explicitly acknowledge each before continuing. Treat "no HTTP/config change in this diff" as the only reason to skip.

## Validation gates for DATA / EXTRACTION changes (must-not-omit — learned the hard way)

Unit-green is necessary, NOT sufficient for pipeline changes. Before claiming an extraction/scanner/resolver/gate change works:

- [ ] **Run a LIVE smoke on a real book** (`extract-entities` on PHB), then inspect the actual output — type counts, names, AND the reject/`declined.json` audit — before trusting or ingesting. (A green-tested gate was structurally DEAD in production this cycle; only the live run caught it.)
- [ ] **Clear the conversion cache after a converter change** (`docker exec … rm -f /books/conversion-cache/*.mineru.json`) — else the smoke runs the OLD mapping. A "cache hit" log at run start when you changed the converter means your fix is NOT being tested.
- [ ] **Early spot-check BEFORE the full ~8 h run.** The first progress checkpoint (`<slug>.progress.json`, ~first 100 candidates / ~15 min) covers front-of-book entities (races, classes). Spot-check the entities your change TARGETS there first — 790 green tests still shipped a Dragonborn→Monster regression that only the live early-check caught.
- [ ] **A count change is NOT automatically a regression — diff WHICH entities changed type.** Monster 34→30 this cycle was a CORRECTION (Acolyte/Noble/Sage/Soldier were mis-typed backgrounds), not a loss. Compare before/after type-maps per entity; never assume a drop is bad or a rise is good.
- [ ] **To pin a SILENT/untraced loss, build a CHEAP deterministic-path harness FIRST — don't burn a ~5-8 h live run to test a hypothesis.** Feed the real inputs (from the conversion cache) through the pipeline stages (matcher → resolver → drop-filter → dedup, and the real `TocCategoryMap` from bookmarks) in a unit test to locate the drop. This session it REFUTED two wrong page→category theories in minutes; the real cause was mis-tagged stat-line headings overwriting the section title. And whenever a valid item can vanish with no entity/decline/error/log, ADD a log at that silent path (the traceless drop cost a full session of diagnosis).
- [ ] **Read the real data shape before writing a predicate over it.** `EntityCandidate.TypePrior` always carries the scanner's frequency floor `{Monster,Spell,Item,Class}` — an "all priors gated" test could never fire. Check the producer (`HeadingCategoryClassifier.ExpandPrior`, the scanner) first.
- [ ] **When recall looks low, separate OUR-logic bugs from UPSTREAM/parser gaps.** If a missing entity is NOT in `declined.json`, it was never a candidate → upstream (Marker/scanner) gap, not our gate. (8 missing classes / 8 missing races this cycle were a parser gap → `mem:project_parser_upgrade_mineru`, not a false-drop.)
- [ ] **A reviewer "verified against the diff" ≠ behavioral truth.** Confirm against live code/producer and real output.
- [ ] **A spec invariant that spans MULTIPLE code paths must be traced across ALL of them at the whole-branch review — per-task green never proves it.** When the spec says "X is excluded from the store" / "Y never happens on any write path" / "Z holds for every caller", each per-task reviewer sees only its own path and passes; the invariant fails only end-to-end. This cycle "`EntityDisposition.Ungrounded` is excluded from `dnd_entities`" was implemented on 2 of 3 write paths (ingest skip + reground delete) but the SHARED `ReindexEntityAsync` (hit by `NeedsReviewService` accept/edit/bulk-accept) still re-indexed it — 8 per-task-green reviews missed it; only the opus whole-branch review, tracing the invariant across every writer, caught it. **Enumerate the write/read paths for any always/never invariant and check each; put the guard at the shared chokepoint, not per-caller.** (Same class: a judge "cannot decide" must map to `Uncertain`, not a false negative — an interface returning `bool` conflated "no" with "call failed"; use a tri-state `bool?`/enum when abstain is semantically distinct.)
- [ ] **Unit fakes for the store/LLM don't prove infra-dependent correctness — add a Testcontainer integration test when a filter/query/parse depends on REAL Qdrant or a REAL model reply.** This repo HAS `Testcontainers.PostgreSql` + `Testcontainers.Qdrant` (`Persistence/PostgresFixture.cs` / `VectorStore/Entities/QdrantFixture.cs`, + Respawn) — REUSE them. A fake `IQdrantSearchClient` proves your filter is *constructed* right, not that Qdrant *matches* on it; a fake `IChatClient` proves your parse handles a canned reply, not the model's real one. Don't let "full suite green (with fakes)" read as "validated against real infra." **And a SELF-SEEDED integration test proves your CODE, not that the REAL corpus has the data.** An integration test that INSERTS the exact fields your feature reads (seeding a `Subclass` entity carrying `hd`/`classFeatures`) only exercises your parse/filter — it cannot tell you the real corpus is missing them. Before trusting green tests for a feature that consumes specific STRUCTURED entity fields, spot-check `books/canonical/*.json` (or hit the real retrieval endpoint) that those fields are actually POPULATED corpus-wide. This cycle the level-up delta read `ClassFields.hd`/`classFeatures`; unit fakes AND a self-seeded Qdrant test were both green, but the real canonical corpus has ~0 Class entities with those fields (the structured-entity layer is thin/unreliable — `mem:project_entity_extraction_rethink`), so HP silently fell back to d8 — caught only by the live smoke + a whole-branch review that COUNTED the canonical files. Fix pattern: skip/mark-unknown rather than fabricate a default when the grounding field is absent (same grounding contract as name-match-or-skip).
- [ ] **A feature that FILTERS on an exact payload-field VALUE (keyword match) must verify its hardcoded key CONSTANTS equal the REAL corpus payload — a self-seeded test passes with the EXPECTED value even when production stores a DIFFERENT one.** Distinct from "is the field populated": here the field IS populated, your filter IS correct, the integration test IS green — but the constant you match against is WRONG for the real data, so it silently matches ZERO. This cycle (`setting-aware-lore`): `SettingCatalog` scoped by source-book keys `PHB`/`MM`/`ERLW`, and the real-Qdrant non-vacuity test seeded blocks with `source_book:"ERLW"` and passed — but `dnd_blocks` actually stores `source_book` as the book DISPLAY NAME (`"Monster Manual 2014"`, `"Eberron: Rising from the Last War"`), so `sourceBook=MM` matched 0 blocks; the whole feature would have returned honest-empty for everything. Caught only by probing the LIVE corpus (`GET /retrieval/search` → inspect `metadata.sourceBook`; a filter-probe `?sourceBook=<key>` returning 0 is the tell). Before trusting a value-filter feature: hit the real endpoint and read back the ACTUAL payload value your constant must equal, then fix the constants (and update the unit test to the real values). Bake this check into the plan as an explicit verification step (it was Task 7 Step 2 here) — the exact-value dependency is invisible to a self-seeded test by construction.
- [ ] **An in-place canonical rewrite (any tool/transform that edits `<slug>.json`) MUST end with a unique-id invariant + a load round-trip.** `CanonicalJsonLoader` THROWS on a duplicate id, so a rewrite that can map two entities onto the same id writes an un-loadable, un-ingestable file. Assert `ids.Should().OnlyHaveUniqueItems()` in a test AND reload the written file (or hit a real read/flag endpoint) before trusting it. This cycle a grounded-vs-grounded name collision wrote 6 duplicate ids that FOUR per-task-green reviews missed — only the whole-branch review + a real-data round-trip caught it. Reuse the extractor's own `EntityNameMatcher` + `EntityIdSlug` so the rewrite ≡ a re-extract (no second stripping/slug impl to drift), and prefer flag-`NeedsReview`-not-delete for ambiguous duplicates.

## Validation gates for PRESENTATIONAL / UI changes (CSS, Blazor markup/classes — learned the hard way)

Unit-green is necessary but **cannot see the UI**. For a CSS / markup-class / styling change, `dotnet build` + `dotnet test` prove only that *behavior* is unchanged — they never render a pixel. Every real defect this cycle (`visual-design-system`) was invisible to build/test and only surfaced by running the app and looking:

- [ ] **Behavior-unchanged proof: build 0/0 AND the FULL `dotnet test` suite stays GREEN.** A presentational change must not alter behavior; a suite that was 1186/1186 must still be 1186/1186. If a "styling" change moves a test, it wasn't presentational — stop and reconcile. (Markup edits allowed in a UI change are class names, thin wrappers, and pure display helpers — e.g. an `HpColor(c)` that returns a token name, a view-state toggle field for a popover. No handler/state/data-flow change; the existing handlers are reused verbatim.)
- [ ] **Every class the markup references MUST resolve to a rule.** Grep the classes used across `CompanionUI/**/*.razor` against `wwwroot/app.css`; a referenced-but-undefined class renders as RAW default HTML (naked inputs/buttons). This cycle whole components (initiative tracker, encounter panel, campaign log) shipped referencing classes that never existed in the stylesheet — invisible to build/test, glaring on screen.
- [ ] **LIVE screenshots are the UI's per-task reviewer — a diff-reviewer can't judge visuals.** FIRST rebuild + restart the app container so you aren't screenshotting pre-commit code — `docker compose build app && docker compose up -d app` (equivalently `docker compose up -d --build app`). The image does NOT hot-reload; a container older than your latest commit silently renders the OLD markup and every screenshot is a false pass (bit this cycle — the running container was 56 min stale). THEN Playwright-screenshot every touched surface. **Capture inline** (`browser_take_screenshot` with `type:"jpeg"` and NO `filename` → the image returns in the tool result; a `filename` saves to the Playwright server's own env, which you can't Read). Log in with the seeded `test`/`test`; the container restart drops the auth cookie, so re-login each rebuild.
- [ ] **Screenshot at BOTH desktop (~1280) AND mobile (~390) width.** The mobile-overflow class of bug is invisible at desktop width and in a diff. After resizing, run a `browser_evaluate` overflow check — `document.documentElement.scrollWidth > clientWidth` means the body scrolls sideways (fix it); also check wide rows (`el.scrollWidth > el.clientWidth`) for internal overflow. This cycle the collapsed mobile top-bar scrolled horizontally (sidebar footer overflow) — caught only by the evaluate check, never by desktop screenshots.
- [ ] **The final diff-review covers CODE-correctness, not looks** (you already judged looks via screenshots): markup edits are behavior-neutral (no logic slipped into a "class-only" change), CSS specificity/cascade has no foot-guns (a broad `input` selector reaching a checkbox; `min-width` defeating `max-width`; element-vs-class fights over padding), and the decorative display font does NOT leak past its allowed spots (wordmark / `h1` / one hero element) into labels/data/buttons.
- **Self-hosted fonts (offline-capable):** `npm pack @fontsource/<family>` (registry.npmjs.org is allowlisted) yields `package/files/<family>-latin-<weight>-normal.woff2`; copy the used weights into `wwwroot/fonts/` + `@font-face`. No CDN → renders offline; published with `wwwroot` static assets.

## Behavior-preserving refactor gates (dedup, moves, EF mapping, god-file splits — learned this cycle)

Structural changes that must not alter behavior have their OWN gates beyond "tests pass":

- [ ] **A shared-TYPE or method-SIGNATURE retype that ripples across callers MUST land ATOMICALLY in ONE task (behavior-preserving), so every task builds 0/0.** Subagent-driven-development commits + builds per task, so a plan that splits "change the model/type" (task N) from "update its callers" (task N+2) leaves RED intermediate commits. Do the WHOLE retype in one task as a behavior-preserving change — same behavior, new type — then add the FEATURE that uses it as a SEPARATE later task on the green base. This cycle: retyping `Combatant.Conditions` from `List<Condition>` → `List<ConditionTimer>` rippled through the helper, `UpdateCombatantAsync`'s param, and the razor's condition uses; the first-draft plan split them (Task 1 model → Task 3 UI) which would have left Tasks 1–2 red. Caught reviewing the PLAN before dispatch and restructured to one atomic retype task (all conditions indefinite, no feature yet) then tick/UI on top. When writing the plan, if a task changes a public type/signature, check who calls it (`find_referencing_symbols`) and fold every caller into that same task.
- [ ] **A feature that DEPENDS on an existing field's INVARIANT (uniqueness / monotonicity / contiguity) MUST verify the PRODUCER guarantees it under all edits — not just the happy path.** A value that's distinct only by luck breaks under deletes/edits, and your new feature is often the FIRST consumer to depend on it, so it surfaces a latent producer weakness rather than introducing one. This cycle: the tie-reorder swaps two combatants' `AddedOrder`, but `AddCombatantAsync` assigned it as `CountAsync(...)` and `RemoveCombatantAsync` never renumbers — so add A=0,B=1,C=2 → remove B → add D=**2** COLLIDES with C; a collided+tied pair makes the swap exchange two EQUAL values → a silent no-op (the ▲/▼ button does nothing). Fix: `max(existing)+1`, never a count/index over a mutable set. When a change relies on a field being unique/ordered, read the PRODUCER (`find_referencing_symbols` on the field's setter) and confirm the invariant holds after a delete, not just a fresh insert. (Sibling of "Read the real data shape before writing a predicate over it" — but about a maintained invariant, not the shape.)
- [ ] **Before collapsing "duplicated" code, prove the copies are IDENTICAL.** If they diverge, PARAMETERIZE the difference or STOP and report — never silently pick one. This cycle: Monster `AlignMap` genuinely diverged (extra `NX`/`NY` keys) → kept local; `ExtractFeatureEntry` differed only by a `minParts` threshold → parameterized; and a "duplicate" `IsSidecar` check was missing `.declined.json` → collapsing to the canonical one FIXED a latent bug (declined files were mis-treated as canonical). Assume divergence until proven identical.
- [ ] **Schema-neutral EF change? PROVE it with an empty migration.** When moving mapping attributes ↔ Fluent config (or any refactor that must not touch the schema), run `dotnet ef migrations add VerifyNeutral`, confirm BOTH `Up()` and `Down()` are empty, then DELETE the migration + restore the model snapshot (`git checkout Migrations/AppDbContextModelSnapshot.cs`). A non-empty migration means your Fluent config diverged from the attributes — fix it, don't ship the drift.
- [ ] **Adding a table? The new migration MUST be ADDITIVE-ONLY, and the model-snapshot diff a PURE insertion.** After `dotnet ef migrations add <Name>`, OPEN the migration: `Up()` creates ONLY the new table + its index, `Down()` drops only it. Read the `AppDbContextModelSnapshot.cs` diff — it must be a pure insertion of the new entity's config; if `Up()` or the snapshot touches ANY OTHER table, the model had drifted from the snapshot and you're silently capturing that drift now (it will alter other tables on the next deploy) — investigate before committing. Commit the migration file AND the updated snapshot together.
- [ ] **A new campaign-scoped (or any parent-owned) table MUST be added to the parent's delete-cascade.** `CampaignRepository.DeleteAsync` cascades `HeroSnapshots → Heroes → Notes → Campaign` by hand (no DB FK — app-level `ExecuteDeleteAsync` inside the execution-strategy transaction). A new child table (e.g. `CampaignLogEntries`, `Combats`) that you DON'T add to that cascade orphans its rows on campaign delete — the whole-branch review caught exactly this. Add a `db.<NewTable>.Where(x => x.CampaignId == id).ExecuteDeleteAsync(...)` line alongside `Notes`, inside the same transaction, and add a test (delete the campaign → the child rows are gone). Same rule for any other parent-scoped delete path. (A child-of-a-child, e.g. `Combatants` under `Combats`, cascades via its DB FK when the parent rows are `ExecuteDelete`d, but delete it explicitly too — child rows before parent — to mirror the `HeroSnapshots→Heroes` ordering and stay FK-safe regardless.)

- [ ] **Adding a case to a discriminated `enum`/`Kind` that a `switch`/`if-else` consumes → update EVERY exhaustive consumer; a catch-all `else`/`default` SILENTLY swallows the new case instead of failing.** During planning, `find_referencing_symbols` on the enum and enumerate each render/format/switch/serialize path as its own task step. This cycle adding `CampaignLogKind.Combat` was missed in `CampaignLog.razor`'s `FormatEntry` (`if (Roll) {…} else {/*Encounter*/}`) — a `Combat` entry fell into the Encounter `else`, deserialized as the wrong payload, found a null field, and was silently dropped (never rendered); only a manual pre-review inspection caught it (build + tests were green — nothing forced the missing branch). Prefer an explicit `else if (Combat)` / exhaustive `switch` whose `default` THROWS over an `else` that absorbs unknowns, and add a render/format test for the new case.
- [ ] **A NEW computation that BRANCHES on an existing domain classification needs a unit test per branch — a single-fixture smoke walks ONE branch only.** When your code reads a classification (`MulticlassSpellcasting.ResolveSlotSource` → full/half/third/**pact**/none; `SpellcastingAbility` → ability-or-null; a level threshold reached/not) and behaves differently per value, a live smoke exercises exactly the ONE case its single hero happens to be, so the special/excluded branch ships untested. Enumerate the classifier's values (`find_symbol` on it) and add a test for EACH — especially the one your code EXCLUDES or treats as absent — and fix by skip/mark-unknown, never a false-positive or fabricated default (the grounding contract, same as name-match-or-skip). Twice this arc a character-advice finding fired wrongly on the excluded branch, both unit-green AND happy-path-smoke-green, both caught only by the opus whole-branch review: slice A fabricated a d8 HP when the structured `hd` field was absent; slice C emitted a false "spell slots don't match" for a single-class Warlock (pact is excluded from the standard multiclass slot table → all-zero computed table vs the sheet's recorded pact slots). The happy-path smoke used a Ranger (half-caster, standard branch) and could not have hit either.
- [ ] **Changing an EXISTING behavior? Every test must DISCRIMINATE old vs new — it must FAIL on the old code and PASS on the new. A test that passes on BOTH gates nothing; a test that encodes the SUPERSEDED behavior must be REWRITTEN, not preserved.** When a change alters (not just adds) behavior, TDD's RED step is the discrimination proof — but confirm the RED is BEHAVIORAL, not a coincidental pass or a mere compile error. This cycle (encounter-swarms, generator re-selection): the `uniform-swarm` test asserted `Count > 1` + `Difficulty == Hard` over a pool of 5 DISTINCT same-CR monsters — which the OLD (remove-each-candidate) algorithm satisfied just as well by picking 5 distinct monsters, so it proved nothing about the new repeat-selection (the reviewer caught it; fix: assert `Monsters.Select(m => m.Id).Distinct().Should().ContainSingle()` so only true repetition passes). Separately, the `sparse-fallback` test encoded the OLD single-monster result and HAD to be rewritten to expect the new 15-copy swarm (an intentional behavior change — flag it as such). And when a PRIOR task already applied the signature/param change your test depends on, the RED shows as a COMPILE error, not a failing assertion — so the behavioral gate never ran; add/keep a test whose assertion (not compilation) fails on the old logic. Rule: for a behavior change, name the OLD result and the NEW result, and make each test flip between them.
- [ ] **Splitting a god file? A strong output-asserting test suite IS the "identical output before/after" guarantee.** Keep it 100% green and NEVER weaken an assertion to make the split pass; only ctor/DI wiring in the tests may change (every assertion stays byte-identical). If no such suite exists, add characterization tests FIRST. (STR-09 split a 657→381-line orchestrator behind its 17-test output suite with zero assertion changes.)
- [ ] **A tool's caller-identity comes from the trusted session, not a spoofable argument.** A tool acting on a user's data must CLOSE OVER the authenticated session's user id (server-verified ownership chain), never accept the id as a tool argument crossing a shared-key/loopback boundary where any key holder could assert it. Ship a NEGATIVE test (caller A cannot touch caller B's data → throws). (SEC-08: `resolve_character_feature` moved off the shared-key MCP surface into a per-user in-process tool.) **A NEW per-user chat tool joins TWO guard tests, not one — and a FILTER-then-assert test is VACUOUS for a tool that isn't in the filter or isn't registered.** The per-user set (`resolve_character_feature`/`check_multiclass`/`rate_encounter`/`build_encounter`/`plan_level_up`/`recommend_build`…) is guarded by (a) a `DndChatServiceTests` "no tool schema exposes a `userId` argument" test that FILTERS the tool list by name then asserts the property, and (b) authenticated-present / unauthenticated-absent presence tests. Adding a tool to only (a) means the schema check runs, but if you forget to register the tool the `Where(name==…)` yields nothing and the assertion passes VACUOUSLY — the security guard silently covers nothing. So a new per-user tool MUST be added to the (a) name filter AND the (b) present/absent lists; a filter-then-assert alone guards the tool's PROPERTY, never its PRESENCE. (Bit `recommend_build` this cycle — the final review caught the vacuous-guard gap.)
- [ ] **New service that a DI-registered consumer depends on → the consumer's `Add*` group MUST pull in the dependency's `Add*`, and the FINAL verify runs the FULL suite, never a feature-filtered subset.** Feature-filtered per-task runs (`dotnet test --filter ~Encounter`) can't match cross-cutting tests — DI/container-graph validation (`FullContainerScopeValidationTests`), composition, scope checks — because those aren't named after your feature. This cycle: every `~Encounter`/`~Chat` task run was GREEN while wiring `DndChatService` (registered by `AddDndChat`) to depend on the new `EncounterDesignService` silently broke the whole-graph validation (a manual test composition never called `AddEncounters()`); only the controller's authoritative FULL `dotnet test` at final verify caught it (1109/1110). FIX: make `AddDndChat` call `AddEncounters()` so no composition can register the consumer without its deps (and drop any now-redundant standalone call) — dependency travels with the consumer, no drift. **And: a NEW `Add*` extension must be registered in TWO places — `Program.cs` AND `FullContainerScopeValidationTests.BuildServiceCollection`, which HAND-REPLICATES Program's `Add*` sequence rather than invoking Program.** Miss the test replica and the scope gate passes VACUOUSLY: the new service is simply absent from the validated container, so a wrong lifetime (e.g. a Singleton consuming the Scoped `DiceRoller`) goes uncaught and the test stays green meaning nothing. This cycle `AddCombat()` was wired into Program but not the replica; the review flagged the gap, and the fix was one line — `services.AddCombat();` in the test's `BuildServiceCollection`, in Program's order.
- [ ] **An ORCHESTRATION/composition feature that threads ONE input to several sub-services must check each sub-service's SEMANTIC interpretation of that input — a value that's fine for one can mean something different (and mismatch) in another.** Same type, different meaning. This cycle (`session-prep`): the tool passed the narrative `theme` ("Sharn intrigue") to all three sub-services, but `EncounterDesignService`'s `theme` is a MONSTER-KEYWORD filter — a narrative theme matches 0 monsters → an empty encounter piece → the LLM hallucinated fake monsters. Unit tests passed (the test's pool monster-source ignores `theme`), so ONLY the live smoke caught it. Fix: pass `theme:null` to the encounter (a party-appropriate baseline the persona re-skins), keep `theme` for the NPC concept + lore question where it's genuinely narrative. When composing, read each sub-service's param DOC/usage (what does IT do with this arg?), and add an integration/live check that exercises the real sub-service (a substitute that ignores the arg hides the mismatch). Sibling of "read the producer's data shape" — but about a param's SEMANTICS across a composition, not its shape.
- [ ] **An OPTIONAL chat-tool parameter MUST have a C# default (`= null`), and its unit test MUST OMIT the key — not pass it as an explicit null.** `AIFunctionFactory` marks every parameter WITHOUT a C# default as `required` in the generated JSON schema; when the LLM then omits that key (which it does for the branch it isn't using — a nonmagical `calculate_crafting` call sends only `{marketValue}` and omits `rarity`/`crafters`), MEAI's binder throws `ArgumentException: The arguments dictionary is missing a value for the required parameter '<name>'` BEFORE the delegate runs. The model narrates this as a vague "function error" and then FABRICATES the answer — exactly the bad math a deterministic tool was built to kill. This is invisible to a unit test whose `ToArgs(new { marketValue = 1500, rarity = (string?)null, crafters = (int?)null })` supplies EVERY key (the null-valued keys satisfy the required-binder), so the test is green while the live tool is broken. FIX: give every optional tool param a `= null` (or real) default so it's optional in the schema and missing keys bind cleanly; and write at least one test that invokes with ONLY the keys the LLM actually sends (`ToArgs(new { marketValue = 1500 })`) — prove it FAILS on the no-default signature and passes with defaults. (Sibling of the prep_session "4-param tools fail MEAI binding" note — now root-caused: it's missing-required-arg on omitted optional keys, not param count per se.) **Companion persona lever:** for a deterministic-math tool to ever be reached, the persona must ROUTE numeric questions to it ("for crafting/encounter/slot MATH you MUST call the calculator tool and report its exact numbers; never compute or infer them") — without that, qwen3 defaults to retrieval tools for any rules-ish question and never selects the calculator. Both fixes are only provable by a LIVE re-smoke after `docker compose up -d --build app` (the persona is baked into the image; config is not hot-reloaded).
- [ ] **Explicit EF transactions + `EnableRetryOnFailure` = wrap in the execution strategy.** The production `AppDbContext` is registered with `EnableRetryOnFailure` (`ServiceCollectionExtensions.cs`), whose `NpgsqlRetryingExecutionStrategy` THROWS on a raw `db.Database.BeginTransactionAsync()` ("does not support user-initiated transactions"). Either drop the explicit transaction when a single `SaveChangesAsync` already gives atomicity, or wrap the multi-statement work in `db.Database.CreateExecutionStrategy().ExecuteAsync(async () => { … tx … })` (use a FRESH context inside the delegate if it Adds tracked rows, so retries don't double-insert). **The default test context has NO retry strategy, so 856 unit tests stayed green while ingestion failed on the first live run** — this is the canonical "unit-green ≠ prod-correct" trap. Guard it with a test built on a retry-ENABLED factory (`UseNpgsql(cs, o => o.EnableRetryOnFailure())`).

## Extraction pipeline facts

- Canonical source of truth: `books/canonical/<slug>.json`; siblings `<slug>.errors.json` / `.warnings.json` / `.declined.json` (declined = official-book gated non-matches). Checkpoints `<slug>.progress*.json` (deleted on success). Files are **root-owned** (container writes them) — edit via `docker cp` a host copy in. **Escape hatches for parser gaps, in preference order:** (1) for official-book SPELLS, `POST /admin/books/{id}/backfill-spells` — deterministic 5etools backfill, idempotent, gap-only, entities marked `dataSource:"5etools-backfill"` (this closed PHB 355→361); (2) hand-author the entity in the canonical (`mem:project_extraction_recall_fixes`). Both beat a 3rd parser/injector patch. **GOTCHA: `extract-entities?force=true` OVERWRITES the whole canonical → hand-authored AND backfilled entities are LOST. After every force re-extract, re-run `backfill-spells` and re-apply hand-authored entities** (e.g. re-add Gnome).
- Parser = **MinerU as a service** at `mineru:8000` (`MinerUPdfConverter` POSTs `/file_parse`, `-b pipeline -m ocr`), the SOLE `IPdfStructureConverter`; Marker is removed. Conversion disk-cache `books/conversion-cache/*.mineru.json` (`mem:project_parser_upgrade_mineru`).
- **CACHE GOTCHA (must-not-omit):** the conversion cache is keyed by **PDF hash only** — converter-LOGIC changes are NOT reflected. After ANY `MinerUPdfConverter` change, `docker exec … rm -f /books/conversion-cache/*.mineru.json` before re-extracting, or the OLD mapping is silently reused (cost a wasted full run this cycle).
- **Extraction is slow** (~30s/candidate, ~8.5 h/book) — qwen3 runs with thinking ON; `think:false` is a ~4-5× lever, and there is no page-range extract yet (full-book only), so early-checkpoint spot-checks are the fast-feedback substitute.
- Dev container lags `main` — **rebuild the app image** (`docker compose up -d --build app`) for code/config to take effect (`mem:project_dev_container_staleness`).

## Red Flags — STOP

- "Tests pass, so the extraction change works." → Run the live smoke, read the output.
- "My integration test passes against real Qdrant, so it's validated." → A SELF-SEEDED integration test injects the fields it reads — it proves your code, not that the real corpus HAS them. Spot-check `books/canonical/*.json` / the real retrieval endpoint for the structured fields your feature consumes before trusting green.
- "My scoping filter's key is `MM`/`ERLW` and the seeded test passes." → The self-seeded test used YOUR expected value; the real `dnd_blocks.source_book` is the DISPLAY NAME (`"Monster Manual 2014"`, `"Eberron: Rising from the Last War"`), so your filter matches 0 in production. Hit `GET /retrieval/search`, read back the ACTUAL payload value, and a `?sourceBook=<key>` probe returning 0 is the tell — fix the constants to the real values.
- "My composition tool threads one input to all its sub-services and the unit tests pass." → A sub-service may interpret that input DIFFERENTLY (session-prep passed a narrative `theme` to the encounter, whose `theme` is a MONSTER-KEYWORD filter → 0 monsters → empty piece). Substitutes that ignore the arg hide it; only the live smoke caught it. Read each sub-service's use of the arg and add a real-sub-service check.
- "I fixed a value/constant and the touched file's filtered tests are green." → A fix that changes a SHARED value (a catalog constant, an enum member's string, a resolved-set) can break tests in OTHER files that assert on that value. Re-run the FULL suite after ANY value/signature-changing fix, not just the filter you edited. (setting-aware: the DATA-INVARIANT catalog fix updated `SettingCatalogTests` — I ran only that filter, green — but `SettingLoreServiceTests` + the integration test still asserted the old `ERLW`/`PHB` values and went red UNDETECTED until the next feature's implementer hit them. An in-loop fix gets the same full-suite gate as a task.)
- "The predicate looks right." → Read the real `TypePrior`/producer shape first.
- "Recall dropped, the gate is broken." → Check `declined.json` — absent ⇒ parser gap, not the gate.
- "I changed the converter; the smoke will test my fix." → A "cache hit" at run start means it tests the OLD mapping. Clear `*.mineru.json` first.
- "The count dropped — that's a regression." → Diff WHICH entities changed type; it may be a correction (mis-typed entities moving to the right type).
- "I'll patch the injector once more for this one entity." → 3rd patch on one entity = STOP. Hand-author it in the canonical.
- "One more parser rule will reach the last N entities." → When parser iterations PLATEAU (PHB spells: +15, then +5, then the rest OCR-damaged beyond any clean rule), STOP grinding — switch to the authoritative backfill (official books) or hand-authoring. Measure the per-iteration yield; a falling curve is the signal.
- "I added the new `enum`/`Kind` case; the existing switch handles it." → A catch-all `else`/`default` SWALLOWS the new case silently (build + tests stay green). `find_referencing_symbols` on the enum, update every consumer, make the default THROW, and test the new case renders.
- "I wired the new `Add*` into Program, so the scope test covers it." → `FullContainerScopeValidationTests` replicates Program's `Add*` list, it doesn't call Program — add the new `Add*` to its `BuildServiceCollection` too, or the gate passes without the service in the graph.
- "The smoke passed, so the finding logic is covered." → A single-fixture smoke walks ONE branch of any per-classification computation (caster type, non-caster, absent field, unreached level). Enumerate the classifier's values and unit-test EACH — especially the excluded/special one (pact/none) — and skip rather than false-positive on it. (Slice-A d8-fabrication and slice-C Warlock false slot-finding both passed unit + Ranger smoke; only the whole-branch review caught them.)
- "My new test passes, so it gates the change." → For a behavior CHANGE, a test that also passes on the OLD code proves nothing. Confirm it FAILS on the old logic (behaviorally, not just a compile error), and REWRITE any test that encodes the superseded behavior. (encounter-swarms: the uniform-swarm test passed under the old remove-each algorithm until `.Distinct().ContainSingle()` forced true repetition; the sparse test had to be rewritten from 1 monster to a 15-copy swarm.)
- "Tests pass, so the UI/CSS change works." → build/test never render a pixel. Screenshot the running app (desktop AND mobile), grep every `.razor` class against `app.css`, and `browser_evaluate` the horizontal overflow. Keep the full suite green to prove behavior didn't change.
- "My `@variable` renders literally in the Razor markup (e.g. `d@die` shows as text)." → a bare `text@identifier` hits Razor's email-address heuristic and is emitted verbatim. Use the explicit expression `text@(identifier)`.
- "It builds, so it's done." → Done = all gates green, output seen.
- "It's a pure dedup." → Prove the copies are byte-identical FIRST; divergence hides bugs (or a fix). Diverging → parameterize or stop.
- "Moving attributes to Fluent won't change the schema." → Prove it: an empty `dotnet ef migrations add` Up/Down, then discard the migration.
- "The god-file split passes tests." → Only if the suite asserts real OUTPUT and no assertion was weakened; wiring-only test edits.
- "The tool takes a userId argument." → Never trust a spoofable id across a shared-key boundary; close over the session identity + ship a negative-ownership test.
- "I'll edit the `.cs` directly." → Serena only.
- "I'll branch for this." → No. Single-dev, work on `main`.
- "It's a one-time data cleanup — I'll add an endpoint." → A one-shot migration is a `Tools/` console (like the retired `Tools/SqliteToPostgres`), NOT a permanent HTTP surface. The app's services are self-contained enough to reuse directly (`new EntityNameMatcher(new EntityNameIndex(fivetoolsDir))` needs no DI/DB). Endpoints are for behavior the product calls again.

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| Coding before a design/proposal | Brainstorm → opsx:propose first. |
| Skipping the live validation on data changes | Smoke on PHB, inspect output + declined.json. |
| Predicate over an unread data shape | Read the producer (scanner/classifier) first. |
| Endpoint change without `.http`/insomnia update | Update both in the same commit. |
| Editing `.cs` by blind text replace | Serena `find_symbol`/`replace_symbol_body`. |
| Creating a feature branch | Work on `main` (single-dev). |
| Grinding parser rules past the yield plateau | Official-book spells: `backfill-spells`; otherwise hand-author. |
| Permanent endpoint for a one-time data migration | `Tools/` console reusing the app's services directly. |
| In-place canonical rewrite trusted on unit-green | Assert unique ids + reload the written file before ingesting. |
| Injecting a concrete service the plan sketched | Depend on the interface (`IIngestionTracker`, not `IngestionTracker`) — this codebase injects interfaces exclusively; a concrete dep needs a redundant registration and constructs a second instance. The plan's example code is a starting point, not binding — task review corrects it. |
