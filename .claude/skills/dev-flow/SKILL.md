# Dev Flow (DndMcpAICsharpFun)

## Overview

The repeatable cycle for changing this project: one ASP.NET Core .NET 10 host serving API + MCP + Blazor UI, with the ingestion/extraction â†’ Postgres/Qdrant pipeline. Each change runs the same loop and **lands directly on `main`** (single-dev, no feature branches).

**Core principle:** No change is "done" on assertion. Done = gates run, output seen, all green. For data/extraction changes, done = **validated on a real book**, not just unit-green. Evidence before claims.

## When to Use

- Starting any feature/change/spec.
- Before writing implementation code.
- Before claiming a change is "done", "fixed", "working", "green", or "validated".
- Before committing or archiving an OpenSpec change.

## The Cycle (never skip, never collapse)

1. **Brainstorm** (`superpowers:brainstorming`) â€” full dialogue, one question at a time, propose 2-3 approaches, get explicit approval. Design is captured in the OpenSpec change, never loose `docs/`.
2. **Propose** (`opsx:propose`) â€” create the OpenSpec change in `openspec/changes/<name>/` (proposal, design, specs, tasks).
3. **Plan** (`superpowers:writing-plans`) â€” detailed bite-sized plan saved to `openspec/changes/<name>/plan.md`.
4. **Implement** (`superpowers:subagent-driven-development`) â€” fresh implementer + reviewer subagent per task, per-task TDD, fix loop on Critical/Important, final whole-branch review on the most capable model. **All on `main`** â€” commit each reviewed task straight to main (commit autonomy granted; `mem:workflow/work_on_main`).
5. **Finish** (user says "commit" / "finish X" on a done spec): **commit â†’ `openspec archive <name> -y` â†’ run `skill-optimizer` â†’ refresh the roadmap memory** (`mem:workflow/finishing_a_spec`). **`ingest-entities` is extraction/canonical-only** â€” run it as the last step when the change adds/edits `books/canonical/*` entities (to project them into `dnd_entities`); retrieval-only, refactor, or endpoint changes SKIP it.

## Serena is MANDATORY for code

All `.cs` reads/edits go through Serena MCP (`find_symbol`, `replace_symbol_body`, `replace_content`, `find_referencing_symbols`). Built-in Read/Edit/Write on code files is **forbidden** (`mem:workflow/serena_first`). Every implementer/reviewer subagent prompt must include the CRITICAL-Serena block + an `initial_instructions` call.

## Quality Gates (must be GREEN before "done")

```bash
dotnet build                                          # 0 warnings (warnings-as-errors via Directory.Build.props)
dotnet test --filter "FullyQualifiedName!~Persistence"  # non-persistence suite (no DB needed)
dotnet test                                           # FULL suite â€” needs Docker (Testcontainers postgres:18)
dotnet format --verify-no-changes                     # formatting/analyzers
pnpm lint:md                                          # markdown: 0 error(s) (after pnpm lint:md:fix); see /lint-md
```

- Persistence tests need Docker running (Testcontainers + Respawn). Non-persistence suite needs nothing.
- Endpoint change (`MapGet/Post/Put/Delete`) â†’ update `DndMcpAICsharpFun.http` AND `dnd-mcp-api.insomnia.json` in the same commit.
- **SECURITY REVIEW (HTTP / endpoint / config changes) â€” mandatory.** The `security-guidance` plugin runs pattern warnings on every edit and an async LLM review on `git commit`/`git push`. Do NOT dismiss it. For any HTTP endpoint (`MapGet/Post/Put/Delete`), request-body/query/header handling, admin-key/auth surface, MCP tool surface, or `Config/appsettings.*`/env/`docker-compose*` change, explicitly review the diff for: **authz/admin-gate gaps, injection/SSRF, secret exposure, and unsafe config/CORS/rate-limit**. If the commit/push hook re-wakes with findings, address or explicitly acknowledge each before continuing. Treat "no HTTP/config change in this diff" as the only reason to skip.

## Validation gates for DATA / EXTRACTION changes (must-not-omit â€” learned the hard way)

Unit-green is necessary, NOT sufficient for pipeline changes. Before claiming an extraction/scanner/resolver/gate change works:

- [ ] **Run a LIVE smoke on a real book** (`extract-entities` on PHB), then inspect the actual output â€” type counts, names, AND the reject/`declined.json` audit â€” before trusting or ingesting. (A green-tested gate was structurally DEAD in production this cycle; only the live run caught it.)
- [ ] **Clear the conversion cache after a converter change** (`docker exec â€¦ rm -f /books/conversion-cache/*.mineru.json`) â€” else the smoke runs the OLD mapping. A "cache hit" log at run start when you changed the converter means your fix is NOT being tested.
- [ ] **Early spot-check BEFORE the full ~8 h run.** The first progress checkpoint (`<slug>.progress.json`, ~first 100 candidates / ~15 min) covers front-of-book entities (races, classes). Spot-check the entities your change TARGETS there first â€” 790 green tests still shipped a Dragonbornâ†’Monster regression that only the live early-check caught.
- [ ] **A count change is NOT automatically a regression â€” diff WHICH entities changed type.** Monster 34â†’30 this cycle was a CORRECTION (Acolyte/Noble/Sage/Soldier were mis-typed backgrounds), not a loss. Compare before/after type-maps per entity; never assume a drop is bad or a rise is good.
- [ ] **To pin a SILENT/untraced loss, build a CHEAP deterministic-path harness FIRST â€” don't burn a ~5-8 h live run to test a hypothesis.** Feed the real inputs (from the conversion cache) through the pipeline stages (matcher â†’ resolver â†’ drop-filter â†’ dedup, and the real `TocCategoryMap` from bookmarks) in a unit test to locate the drop. This session it REFUTED two wrong pageâ†’category theories in minutes; the real cause was mis-tagged stat-line headings overwriting the section title. And whenever a valid item can vanish with no entity/decline/error/log, ADD a log at that silent path (the traceless drop cost a full session of diagnosis).
- [ ] **Read the real data shape before writing a predicate over it.** `EntityCandidate.TypePrior` always carries the scanner's frequency floor `{Monster,Spell,Item,Class}` â€” an "all priors gated" test could never fire. Check the producer (`HeadingCategoryClassifier.ExpandPrior`, the scanner) first.
- [ ] **When recall looks low, separate OUR-logic bugs from UPSTREAM/parser gaps.** If a missing entity is NOT in `declined.json`, it was never a candidate â†’ upstream (Marker/scanner) gap, not our gate. (8 missing classes / 8 missing races this cycle were a parser gap â†’ `mem:project_parser_upgrade_mineru`, not a false-drop.)
- [ ] **A reviewer "verified against the diff" â‰  behavioral truth.** Confirm against live code/producer and real output.
- [ ] **A spec invariant that spans MULTIPLE code paths must be traced across ALL of them at the whole-branch review â€” per-task green never proves it.** When the spec says "X is excluded from the store" / "Y never happens on any write path" / "Z holds for every caller", each per-task reviewer sees only its own path and passes; the invariant fails only end-to-end. This cycle "`EntityDisposition.Ungrounded` is excluded from `dnd_entities`" was implemented on 2 of 3 write paths (ingest skip + reground delete) but the SHARED `ReindexEntityAsync` (hit by `NeedsReviewService` accept/edit/bulk-accept) still re-indexed it â€” 8 per-task-green reviews missed it; only the opus whole-branch review, tracing the invariant across every writer, caught it. **Enumerate the write/read paths for any always/never invariant and check each; put the guard at the shared chokepoint, not per-caller.** (Same class: a judge "cannot decide" must map to `Uncertain`, not a false negative â€” an interface returning `bool` conflated "no" with "call failed"; use a tri-state `bool?`/enum when abstain is semantically distinct.)
- [ ] **Unit fakes for the store/LLM don't prove infra-dependent correctness â€” add a Testcontainer integration test when a filter/query/parse depends on REAL Qdrant or a REAL model reply.** This repo HAS `Testcontainers.PostgreSql` + `Testcontainers.Qdrant` (`Persistence/PostgresFixture.cs` / `VectorStore/Entities/QdrantFixture.cs`, + Respawn) â€” REUSE them. A fake `IQdrantSearchClient` proves your filter is *constructed* right, not that Qdrant *matches* on it; a fake `IChatClient` proves your parse handles a canned reply, not the model's real one. Don't let "full suite green (with fakes)" read as "validated against real infra."
- [ ] **An in-place canonical rewrite (any tool/transform that edits `<slug>.json`) MUST end with a unique-id invariant + a load round-trip.** `CanonicalJsonLoader` THROWS on a duplicate id, so a rewrite that can map two entities onto the same id writes an un-loadable, un-ingestable file. Assert `ids.Should().OnlyHaveUniqueItems()` in a test AND reload the written file (or hit a real read/flag endpoint) before trusting it. This cycle a grounded-vs-grounded name collision wrote 6 duplicate ids that FOUR per-task-green reviews missed â€” only the whole-branch review + a real-data round-trip caught it. Reuse the extractor's own `EntityNameMatcher` + `EntityIdSlug` so the rewrite â‰ˇ a re-extract (no second stripping/slug impl to drift), and prefer flag-`NeedsReview`-not-delete for ambiguous duplicates.

## Validation gates for PRESENTATIONAL / UI changes (CSS, Blazor markup/classes â€” learned the hard way)

Unit-green is necessary but **cannot see the UI**. For a CSS / markup-class / styling change, `dotnet build` + `dotnet test` prove only that *behavior* is unchanged â€” they never render a pixel. Every real defect this cycle (`visual-design-system`) was invisible to build/test and only surfaced by running the app and looking:

- [ ] **Behavior-unchanged proof: build 0/0 AND the FULL `dotnet test` suite stays GREEN.** A presentational change must not alter behavior; a suite that was 1186/1186 must still be 1186/1186. If a "styling" change moves a test, it wasn't presentational â€” stop and reconcile. (Markup edits allowed in a UI change are class names, thin wrappers, and pure display helpers â€” e.g. an `HpColor(c)` that returns a token name, a view-state toggle field for a popover. No handler/state/data-flow change; the existing handlers are reused verbatim.)
- [ ] **Every class the markup references MUST resolve to a rule.** Grep the classes used across `CompanionUI/**/*.razor` against `wwwroot/app.css`; a referenced-but-undefined class renders as RAW default HTML (naked inputs/buttons). This cycle whole components (initiative tracker, encounter panel, campaign log) shipped referencing classes that never existed in the stylesheet â€” invisible to build/test, glaring on screen.
- [ ] **LIVE screenshots are the UI's per-task reviewer â€” a diff-reviewer can't judge visuals.** FIRST rebuild + restart the app container so you aren't screenshotting pre-commit code â€” `docker compose build app && docker compose up -d app` (equivalently `docker compose up -d --build app`). The image does NOT hot-reload; a container older than your latest commit silently renders the OLD markup and every screenshot is a false pass (bit this cycle â€” the running container was 56 min stale). THEN Playwright-screenshot every touched surface. **Capture inline** (`browser_take_screenshot` with `type:"jpeg"` and NO `filename` â†’ the image returns in the tool result; a `filename` saves to the Playwright server's own env, which you can't Read). Log in with the seeded `test`/`test`; the container restart drops the auth cookie, so re-login each rebuild.
- [ ] **Screenshot at BOTH desktop (~1280) AND mobile (~390) width.** The mobile-overflow class of bug is invisible at desktop width and in a diff. After resizing, run a `browser_evaluate` overflow check â€” `document.documentElement.scrollWidth > clientWidth` means the body scrolls sideways (fix it); also check wide rows (`el.scrollWidth > el.clientWidth`) for internal overflow. This cycle the collapsed mobile top-bar scrolled horizontally (sidebar footer overflow) â€” caught only by the evaluate check, never by desktop screenshots.
- [ ] **The final diff-review covers CODE-correctness, not looks** (you already judged looks via screenshots): markup edits are behavior-neutral (no logic slipped into a "class-only" change), CSS specificity/cascade has no foot-guns (a broad `input` selector reaching a checkbox; `min-width` defeating `max-width`; element-vs-class fights over padding), and the decorative display font does NOT leak past its allowed spots (wordmark / `h1` / one hero element) into labels/data/buttons.
- **Self-hosted fonts (offline-capable):** `npm pack @fontsource/<family>` (registry.npmjs.org is allowlisted) yields `package/files/<family>-latin-<weight>-normal.woff2`; copy the used weights into `wwwroot/fonts/` + `@font-face`. No CDN â†’ renders offline; published with `wwwroot` static assets.

## Behavior-preserving refactor gates (dedup, moves, EF mapping, god-file splits â€” learned this cycle)

Structural changes that must not alter behavior have their OWN gates beyond "tests pass":

- [ ] **A shared-TYPE or method-SIGNATURE retype that ripples across callers MUST land ATOMICALLY in ONE task (behavior-preserving), so every task builds 0/0.** Subagent-driven-development commits + builds per task, so a plan that splits "change the model/type" (task N) from "update its callers" (task N+2) leaves RED intermediate commits. Do the WHOLE retype in one task as a behavior-preserving change â€” same behavior, new type â€” then add the FEATURE that uses it as a SEPARATE later task on the green base. This cycle: retyping `Combatant.Conditions` from `List<Condition>` â†’ `List<ConditionTimer>` rippled through the helper, `UpdateCombatantAsync`'s param, and the razor's condition uses; the first-draft plan split them (Task 1 model â†’ Task 3 UI) which would have left Tasks 1â€“2 red. Caught reviewing the PLAN before dispatch and restructured to one atomic retype task (all conditions indefinite, no feature yet) then tick/UI on top. When writing the plan, if a task changes a public type/signature, check who calls it (`find_referencing_symbols`) and fold every caller into that same task.
- [ ] **A feature that DEPENDS on an existing field's INVARIANT (uniqueness / monotonicity / contiguity) MUST verify the PRODUCER guarantees it under all edits â€” not just the happy path.** A value that's distinct only by luck breaks under deletes/edits, and your new feature is often the FIRST consumer to depend on it, so it surfaces a latent producer weakness rather than introducing one. This cycle: the tie-reorder swaps two combatants' `AddedOrder`, but `AddCombatantAsync` assigned it as `CountAsync(...)` and `RemoveCombatantAsync` never renumbers â€” so add A=0,B=1,C=2 â†’ remove B â†’ add D=**2** COLLIDES with C; a collided+tied pair makes the swap exchange two EQUAL values â†’ a silent no-op (the â–˛/â–Ľ button does nothing). Fix: `max(existing)+1`, never a count/index over a mutable set. When a change relies on a field being unique/ordered, read the PRODUCER (`find_referencing_symbols` on the field's setter) and confirm the invariant holds after a delete, not just a fresh insert. (Sibling of "Read the real data shape before writing a predicate over it" â€” but about a maintained invariant, not the shape.)
- [ ] **Before collapsing "duplicated" code, prove the copies are IDENTICAL.** If they diverge, PARAMETERIZE the difference or STOP and report â€” never silently pick one. This cycle: Monster `AlignMap` genuinely diverged (extra `NX`/`NY` keys) â†’ kept local; `ExtractFeatureEntry` differed only by a `minParts` threshold â†’ parameterized; and a "duplicate" `IsSidecar` check was missing `.declined.json` â†’ collapsing to the canonical one FIXED a latent bug (declined files were mis-treated as canonical). Assume divergence until proven identical.
- [ ] **Schema-neutral EF change? PROVE it with an empty migration.** When moving mapping attributes â†” Fluent config (or any refactor that must not touch the schema), run `dotnet ef migrations add VerifyNeutral`, confirm BOTH `Up()` and `Down()` are empty, then DELETE the migration + restore the model snapshot (`git checkout Migrations/AppDbContextModelSnapshot.cs`). A non-empty migration means your Fluent config diverged from the attributes â€” fix it, don't ship the drift.
- [ ] **Adding a table? The new migration MUST be ADDITIVE-ONLY, and the model-snapshot diff a PURE insertion.** After `dotnet ef migrations add <Name>`, OPEN the migration: `Up()` creates ONLY the new table + its index, `Down()` drops only it. Read the `AppDbContextModelSnapshot.cs` diff â€” it must be a pure insertion of the new entity's config; if `Up()` or the snapshot touches ANY OTHER table, the model had drifted from the snapshot and you're silently capturing that drift now (it will alter other tables on the next deploy) â€” investigate before committing. Commit the migration file AND the updated snapshot together.
- [ ] **A new campaign-scoped (or any parent-owned) table MUST be added to the parent's delete-cascade.** `CampaignRepository.DeleteAsync` cascades `HeroSnapshots â†’ Heroes â†’ Notes â†’ Campaign` by hand (no DB FK â€” app-level `ExecuteDeleteAsync` inside the execution-strategy transaction). A new child table (e.g. `CampaignLogEntries`, `Combats`) that you DON'T add to that cascade orphans its rows on campaign delete â€” the whole-branch review caught exactly this. Add a `db.<NewTable>.Where(x => x.CampaignId == id).ExecuteDeleteAsync(...)` line alongside `Notes`, inside the same transaction, and add a test (delete the campaign â†’ the child rows are gone). Same rule for any other parent-scoped delete path. (A child-of-a-child, e.g. `Combatants` under `Combats`, cascades via its DB FK when the parent rows are `ExecuteDelete`d, but delete it explicitly too â€” child rows before parent â€” to mirror the `HeroSnapshotsâ†’Heroes` ordering and stay FK-safe regardless.)

- [ ] **Adding a case to a discriminated `enum`/`Kind` that a `switch`/`if-else` consumes â†’ update EVERY exhaustive consumer; a catch-all `else`/`default` SILENTLY swallows the new case instead of failing.** During planning, `find_referencing_symbols` on the enum and enumerate each render/format/switch/serialize path as its own task step. This cycle adding `CampaignLogKind.Combat` was missed in `CampaignLog.razor`'s `FormatEntry` (`if (Roll) {â€¦} else {/*Encounter*/}`) â€” a `Combat` entry fell into the Encounter `else`, deserialized as the wrong payload, found a null field, and was silently dropped (never rendered); only a manual pre-review inspection caught it (build + tests were green â€” nothing forced the missing branch). Prefer an explicit `else if (Combat)` / exhaustive `switch` whose `default` THROWS over an `else` that absorbs unknowns, and add a render/format test for the new case.
- [ ] **Splitting a god file? A strong output-asserting test suite IS the "identical output before/after" guarantee.** Keep it 100% green and NEVER weaken an assertion to make the split pass; only ctor/DI wiring in the tests may change (every assertion stays byte-identical). If no such suite exists, add characterization tests FIRST. (STR-09 split a 657â†’381-line orchestrator behind its 17-test output suite with zero assertion changes.)
- [ ] **A tool's caller-identity comes from the trusted session, not a spoofable argument.** A tool acting on a user's data must CLOSE OVER the authenticated session's user id (server-verified ownership chain), never accept the id as a tool argument crossing a shared-key/loopback boundary where any key holder could assert it. Ship a NEGATIVE test (caller A cannot touch caller B's data â†’ throws). (SEC-08: `resolve_character_feature` moved off the shared-key MCP surface into a per-user in-process tool.)
- [ ] **New service that a DI-registered consumer depends on â†’ the consumer's `Add*` group MUST pull in the dependency's `Add*`, and the FINAL verify runs the FULL suite, never a feature-filtered subset.** Feature-filtered per-task runs (`dotnet test --filter ~Encounter`) can't match cross-cutting tests â€” DI/container-graph validation (`FullContainerScopeValidationTests`), composition, scope checks â€” because those aren't named after your feature. This cycle: every `~Encounter`/`~Chat` task run was GREEN while wiring `DndChatService` (registered by `AddDndChat`) to depend on the new `EncounterDesignService` silently broke the whole-graph validation (a manual test composition never called `AddEncounters()`); only the controller's authoritative FULL `dotnet test` at final verify caught it (1109/1110). FIX: make `AddDndChat` call `AddEncounters()` so no composition can register the consumer without its deps (and drop any now-redundant standalone call) â€” dependency travels with the consumer, no drift. **And: a NEW `Add*` extension must be registered in TWO places â€” `Program.cs` AND `FullContainerScopeValidationTests.BuildServiceCollection`, which HAND-REPLICATES Program's `Add*` sequence rather than invoking Program.** Miss the test replica and the scope gate passes VACUOUSLY: the new service is simply absent from the validated container, so a wrong lifetime (e.g. a Singleton consuming the Scoped `DiceRoller`) goes uncaught and the test stays green meaning nothing. This cycle `AddCombat()` was wired into Program but not the replica; the review flagged the gap, and the fix was one line â€” `services.AddCombat();` in the test's `BuildServiceCollection`, in Program's order.
- [ ] **Explicit EF transactions + `EnableRetryOnFailure` = wrap in the execution strategy.** The production `AppDbContext` is registered with `EnableRetryOnFailure` (`ServiceCollectionExtensions.cs`), whose `NpgsqlRetryingExecutionStrategy` THROWS on a raw `db.Database.BeginTransactionAsync()` ("does not support user-initiated transactions"). Either drop the explicit transaction when a single `SaveChangesAsync` already gives atomicity, or wrap the multi-statement work in `db.Database.CreateExecutionStrategy().ExecuteAsync(async () => { â€¦ tx â€¦ })` (use a FRESH context inside the delegate if it Adds tracked rows, so retries don't double-insert). **The default test context has NO retry strategy, so 856 unit tests stayed green while ingestion failed on the first live run** â€” this is the canonical "unit-green â‰  prod-correct" trap. Guard it with a test built on a retry-ENABLED factory (`UseNpgsql(cs, o => o.EnableRetryOnFailure())`).

## Extraction pipeline facts

- Canonical source of truth: `books/canonical/<slug>.json`; siblings `<slug>.errors.json` / `.warnings.json` / `.declined.json` (declined = official-book gated non-matches). Checkpoints `<slug>.progress*.json` (deleted on success). Files are **root-owned** (container writes them) â€” edit via `docker cp` a host copy in. **Escape hatches for parser gaps, in preference order:** (1) for official-book SPELLS, `POST /admin/books/{id}/backfill-spells` â€” deterministic 5etools backfill, idempotent, gap-only, entities marked `dataSource:"5etools-backfill"` (this closed PHB 355â†’361); (2) hand-author the entity in the canonical (`mem:project_extraction_recall_fixes`). Both beat a 3rd parser/injector patch. **GOTCHA: `extract-entities?force=true` OVERWRITES the whole canonical â†’ hand-authored AND backfilled entities are LOST. After every force re-extract, re-run `backfill-spells` and re-apply hand-authored entities** (e.g. re-add Gnome).
- Parser = **MinerU as a service** at `mineru:8000` (`MinerUPdfConverter` POSTs `/file_parse`, `-b pipeline -m ocr`), the SOLE `IPdfStructureConverter`; Marker is removed. Conversion disk-cache `books/conversion-cache/*.mineru.json` (`mem:project_parser_upgrade_mineru`).
- **CACHE GOTCHA (must-not-omit):** the conversion cache is keyed by **PDF hash only** â€” converter-LOGIC changes are NOT reflected. After ANY `MinerUPdfConverter` change, `docker exec â€¦ rm -f /books/conversion-cache/*.mineru.json` before re-extracting, or the OLD mapping is silently reused (cost a wasted full run this cycle).
- **Extraction is slow** (~30s/candidate, ~8.5 h/book) â€” qwen3 runs with thinking ON; `think:false` is a ~4-5Ă— lever, and there is no page-range extract yet (full-book only), so early-checkpoint spot-checks are the fast-feedback substitute.
- Dev container lags `main` â€” **rebuild the app image** (`docker compose up -d --build app`) for code/config to take effect (`mem:project_dev_container_staleness`).

## Red Flags â€” STOP

- "Tests pass, so the extraction change works." â†’ Run the live smoke, read the output.
- "The predicate looks right." â†’ Read the real `TypePrior`/producer shape first.
- "Recall dropped, the gate is broken." â†’ Check `declined.json` â€” absent â‡’ parser gap, not the gate.
- "I changed the converter; the smoke will test my fix." â†’ A "cache hit" at run start means it tests the OLD mapping. Clear `*.mineru.json` first.
- "The count dropped â€” that's a regression." â†’ Diff WHICH entities changed type; it may be a correction (mis-typed entities moving to the right type).
- "I'll patch the injector once more for this one entity." â†’ 3rd patch on one entity = STOP. Hand-author it in the canonical.
- "One more parser rule will reach the last N entities." â†’ When parser iterations PLATEAU (PHB spells: +15, then +5, then the rest OCR-damaged beyond any clean rule), STOP grinding â€” switch to the authoritative backfill (official books) or hand-authoring. Measure the per-iteration yield; a falling curve is the signal.
- "I added the new `enum`/`Kind` case; the existing switch handles it." â†’ A catch-all `else`/`default` SWALLOWS the new case silently (build + tests stay green). `find_referencing_symbols` on the enum, update every consumer, make the default THROW, and test the new case renders.
- "I wired the new `Add*` into Program, so the scope test covers it." â†’ `FullContainerScopeValidationTests` replicates Program's `Add*` list, it doesn't call Program â€” add the new `Add*` to its `BuildServiceCollection` too, or the gate passes without the service in the graph.
- "Tests pass, so the UI/CSS change works." â†’ build/test never render a pixel. Screenshot the running app (desktop AND mobile), grep every `.razor` class against `app.css`, and `browser_evaluate` the horizontal overflow. Keep the full suite green to prove behavior didn't change.
- "My `@variable` renders literally in the Razor markup (e.g. `d@die` shows as text)." â†’ a bare `text@identifier` hits Razor's email-address heuristic and is emitted verbatim. Use the explicit expression `text@(identifier)`.
- "It builds, so it's done." â†’ Done = all gates green, output seen.
- "It's a pure dedup." â†’ Prove the copies are byte-identical FIRST; divergence hides bugs (or a fix). Diverging â†’ parameterize or stop.
- "Moving attributes to Fluent won't change the schema." â†’ Prove it: an empty `dotnet ef migrations add` Up/Down, then discard the migration.
- "The god-file split passes tests." â†’ Only if the suite asserts real OUTPUT and no assertion was weakened; wiring-only test edits.
- "The tool takes a userId argument." â†’ Never trust a spoofable id across a shared-key boundary; close over the session identity + ship a negative-ownership test.
- "I'll edit the `.cs` directly." â†’ Serena only.
- "I'll branch for this." â†’ No. Single-dev, work on `main`.
- "It's a one-time data cleanup â€” I'll add an endpoint." â†’ A one-shot migration is a `Tools/` console (like the retired `Tools/SqliteToPostgres`), NOT a permanent HTTP surface. The app's services are self-contained enough to reuse directly (`new EntityNameMatcher(new EntityNameIndex(fivetoolsDir))` needs no DI/DB). Endpoints are for behavior the product calls again.

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| Coding before a design/proposal | Brainstorm â†’ opsx:propose first. |
| Skipping the live validation on data changes | Smoke on PHB, inspect output + declined.json. |
| Predicate over an unread data shape | Read the producer (scanner/classifier) first. |
| Endpoint change without `.http`/insomnia update | Update both in the same commit. |
| Editing `.cs` by blind text replace | Serena `find_symbol`/`replace_symbol_body`. |
| Creating a feature branch | Work on `main` (single-dev). |
| Grinding parser rules past the yield plateau | Official-book spells: `backfill-spells`; otherwise hand-author. |
| Permanent endpoint for a one-time data migration | `Tools/` console reusing the app's services directly. |
| In-place canonical rewrite trusted on unit-green | Assert unique ids + reload the written file before ingesting. |
| Injecting a concrete service the plan sketched | Depend on the interface (`IIngestionTracker`, not `IngestionTracker`) â€” this codebase injects interfaces exclusively; a concrete dep needs a redundant registration and constructs a second instance. The plan's example code is a starting point, not binding â€” task review corrects it. |
