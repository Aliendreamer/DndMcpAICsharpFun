## Requirements
### Requirement: A single EF Core DbContext owns all persistence
The system SHALL persist all relational data through one EF Core `AppDbContext` containing `DbSet`s for ingestion records and the user/campaign/hero/snapshot (and chat, if persisted) entities. The standalone `IngestionDbContext` SHALL be replaced by this unified context. Persistence SHALL remain on SQLite in this change.

#### Scenario: One context covers all tables
- **WHEN** the application starts and applies migrations
- **THEN** a single `AppDbContext` migration creates the ingestion table(s) and the user/campaign/hero/snapshot table(s) in one SQLite database

#### Scenario: No second DbContext
- **WHEN** the codebase is inspected
- **THEN** there is no `IngestionDbContext` and no separate companion DbContext; only `AppDbContext` exists

### Requirement: Domain repositories use EF Core, not raw ADO.NET
The system SHALL implement `UserRepository`, `CampaignRepository`, and `HeroRepository` using EF Core against `AppDbContext`. Hand-written SQL (`CREATE TABLE`, `AUTOINCREMENT`, `last_insert_rowid()`) and direct `SqliteConnection` usage SHALL be removed. The repositories' public method signatures SHALL be preserved so UI pages and services are unaffected.

#### Scenario: Repository methods unchanged for callers
- **WHEN** a Blazor page or service calls an existing repository method
- **THEN** the method signature and returned types are unchanged from the pre-merge implementation

#### Scenario: Identity keys via the provider
- **WHEN** a new user, campaign, or hero is inserted
- **THEN** its identity key is generated by EF Core / the database provider, not by `last_insert_rowid()`

### Requirement: Character sheets persist as JSON
The system SHALL store `HeroSnapshot` character-sheet data as a JSON column via an EF Core value conversion, matching the pre-merge `CharacterJson` storage.

#### Scenario: Snapshot round-trips
- **WHEN** a hero snapshot is saved and later loaded
- **THEN** the deserialized `CharacterSheet` equals what was saved

### Requirement: Chat history is persisted
The system SHALL persist chat conversation turns in a `ChatMessage` table within `AppDbContext`, associated with the relevant user (and hero/campaign context where applicable), recording at least role (user/assistant), content, and timestamp. The chat flow SHALL save each turn and SHALL be able to load prior turns for a conversation. This is net-new behavior; before this change chat was stateless.

#### Scenario: Turns are saved
- **WHEN** a user sends a message and the assistant replies
- **THEN** both the user turn and the assistant turn are stored in the `ChatMessage` table with role, content, and timestamp

#### Scenario: Prior history can be loaded
- **WHEN** a conversation is reopened
- **THEN** previously stored turns for that conversation are retrievable in chronological order

### Requirement: Schema is created via EF migrations at startup
The system SHALL create and update the database schema using EF Core migrations applied at startup, replacing the companion's imperative `CREATE TABLE` initialization.

#### Scenario: Fresh database is migrated on boot
- **WHEN** the application starts against an empty database
- **THEN** the unified `AppDbContext` migration runs and all tables exist before requests are served
