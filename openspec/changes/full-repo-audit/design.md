## Context

~28k LOC of C# (364 files) in one host project (API + MCP server + Blazor Server UI) plus a test project (Testcontainers/Postgres), a schema-generator tool, and Docker/compose infra. Recent work was feature-velocity-driven (extraction pipeline, spell recall); no holistic audit exists. The prod compose was recently hardened (services bound to 127.0.0.1) because the app is heading to an internet-exposed VPS. The user wants a findings-only audit; fixes are triaged together afterwards.

Constraints:

- Read-only over the repo; the deliverable is one report file. No behavior changes.
- git-crypt boundary: `Config/appsettings*.json` and `.env*` contents must never be read (sandbox-masked anyway). Secret *handling* is in scope via code/config references (`IConfiguration` usage, env var names, compose `environment:` blocks).
- Agent fan-out is explicitly authorized by the user for the audit run.
- 28k LOC exceeds a single reviewer context — coverage must be partitioned deterministically, not left to agent discretion.

## Goals / Non-Goals

**Goals:**

- Every audit surface visited with an explicit coverage map: 12 feature slices, `Domain/`, `Infrastructure/`, `CompanionUI/`, `Program.cs`, tests, `Tools/SchemaGenerator`, Dockerfile, both composes, `Directory.*` build files, contract files (`.http`/insomnia).
- Five dimensions per the brainstorm: security (internet-exposed-VPS primary threat model), structure/architecture, .NET best practices, simplification/dead code, correctness (incl. test-quality and infra).
- Every endpoint's auth posture explicitly classified (anonymous / user-auth / admin / MCP-key) in the report — no endpoint unaccounted for.
- Zero unverified Critical/Important findings: each is independently re-derived against live code by a skeptic pass before publication.
- Triage-ready report: stable finding IDs, severity, dimension, `file:line`, what/why, concrete recommendation, S/M/L effort — so fix batches can be lifted straight into follow-up changes.

**Non-Goals:**

- Fixing anything (separate follow-up changes, decided with the user).
- Auditing `books/` data, canonical JSON contents, `openspec/` history, `node_modules/`, generated `Schemas/canonical/*.json`.
- Reading secret values; penetration testing a running deployment; performance benchmarking (static review only).

## Decisions

**D1 — Three-stage pipeline: tooling → parallel dimension audits → adversarial verify → synthesis.**
Automated checks run first (`dotnet list package --vulnerable --include-transitive`, `dotnet build` warning baseline, `dotnet format --verify-no-changes`, scripted grep for secret-pattern leaks / `AllowAnonymous` / disabled-auth markers) because they are cheap, objective, and give the manual passes evidence to build on. Then read-only audit agents fan out (user-authorized) — one per dimension, with the security agent additionally given the endpoint inventory to classify. Then every Critical/Important finding goes to an independent verifier prompted to *refute* it against live code; refuted findings are dropped or downgraded with the refutation noted. Alternative considered: one sequential deep pass (rejected: context limits silently degrade tail coverage); tooling-only (rejected: misses structure/simplification/design issues entirely).

**D2 — Deterministic coverage partitioning.** Each agent receives an explicit file-list manifest (generated by `find`/`ls` at run time), not "audit the repo". Agents must return a per-path visited/skipped ledger; skips are re-dispatched. This is the "no silent caps" rule — coverage is proven, not assumed.

**D3 — Dimension set fixed at five.** Security; structure/architecture (slice boundaries, Domain purity, dependency direction, oversized files); .NET best practices (async correctness, DI lifetimes, EF Core patterns, nullability, disposal, cancellation); simplification (dead code, duplication, over-abstraction, unused packages); correctness+tests+infra (logic bugs, race conditions, test smells/gaps on critical paths, Docker/compose hygiene). Cross-dimension duplicates are merged at synthesis by `file:line` key, keeping the highest-severity framing.

**D4 — Severity and threat model.** Critical = exploitable security flaw under the VPS model or data-loss/corruption bug. Important = likely bug, significant security weakening, or structural problem that actively taxes development. Minor = style, simplification, low-risk best-practice drift. Security findings that only matter when internet-exposed are tagged `[vps]`; ones that hold even locally are untagged.

**D5 — Report is the only artifact; it lives in `docs/audits/`.** `docs/audits/2026-07-02-full-repo-audit.md`, hand-authored-markdown rules apply (`pnpm lint:md` clean). Findings get stable IDs (`SEC-01`, `STR-01`, `NET-01`, `SIM-01`, `COR-01`, …) so triage decisions ("fix SEC-01..03 in one change") are unambiguous. The openspec change dir holds only spec/plan artifacts, so archiving later doesn't bury the report.

**D6 — Verification asymmetry.** Only Critical/Important findings get the full refute pass (they drive fix decisions and must be trustworthy). Minor findings are spot-checked at synthesis; a wrong Minor costs little in triage.

## Risks / Trade-offs

- [Audit agents hallucinate findings] → adversarial verify gate on Critical/Important; every finding must cite `file:line` that the verifier re-reads.
- [Coverage gaps despite manifests] → visited/skipped ledger per agent; synthesis cross-checks the ledger against the manifest and re-dispatches gaps before writing the report.
- [Finding volume overwhelms triage] → severity triage + per-dimension grouping + effort estimates; Minor findings summarized in compact tables.
- [Secret exposure during audit] → agents instructed to never read `Config/appsettings*.json`/`.env*` (sandbox masks them regardless); compose `environment:` values referencing secrets are reported by variable *name* only.
- [Token cost of fan-out] → accepted; user explicitly authorized agent fan-out. Tooling-first ordering keeps manual passes focused.
- [Report goes stale as main moves] → report header records the audited commit SHA; findings cite that snapshot.

## Open Questions
