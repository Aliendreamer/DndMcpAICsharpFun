## Context

`CharacterSheet` is persisted as a JSON string column (`AppDbContext.cs` — the sheet is serialized whole, matching the legacy `CharacterJson` column), so adding a `WornArmor` property needs NO EF migration: old snapshots deserialize it to its default (unarmored). The AC in the hero editor today is a manual `<input type="number" @bind="_editSheet.ArmorClass">` (`HeroDetail.razor:260`); `Equipment` is a free-text list. `CharacterResolutionService.ResolveForSheetAsync` dispatches feature strings to `Resolve…` methods returning `ResolvedFact` with per-component provenance + an `ok`/`needsReview` confidence; `ResolveSavingThrows`/`ResolveMulticlassValidity` are the existing PURE (no-DB) static resolvers. `SavingThrowProficiencies` is the established static-class-data pattern to mirror for `ArmorCatalog`.

## Goals / Non-Goals

**Goals:** structured worn-armor capture + a deterministic, grounded `armor class` resolver (armor + Dex-by-category + shield + magic + Unarmored Defense), consistent with the other resolver features. **Non-Goals:** no EF migration (JSON column); no change to the stored manual `ArmorClass` int (the resolver neither reads nor writes it); no live AC calc in the UI; no custom/homebrew armor entry (unknown armor → `needsReview`, custom deferred); no corpus/Item-entity armor lookup (static catalog only — no grounding risk).

## Decisions

- **D1 — `WornArmor` shape.** `sealed class WornArmor { string ArmorName = ""; bool Shield; int MagicBonus; }` on `CharacterSheet` (JSON). `ArmorName` empty or `"None"` (case-insensitive) = unarmored. Kept minimal — the armor's base AC + category come from the catalog by name, not stored on the sheet (single source of truth; a catalog fix corrects every character).
- **D2 — `ArmorCatalog` (static).** `name → (int BaseAc, ArmorCategory Category)` for the PHB set: Light — Padded 11, Leather 11, Studded Leather 12; Medium — Hide 12, Chain Shirt 13, Scale Mail 14, Breastplate 14, Half Plate 15; Heavy — Ring Mail 14, Chain Mail 16, Splint 17, Plate 18. `enum ArmorCategory { Light, Medium, Heavy }`. Case-insensitive lookup returning a nullable tuple; unknown non-unarmored name → the resolver's `needsReview` path.
- **D3 — AC computation (pure).** Given the sheet: if unarmored (no `ArmorName`), `armorAc = 10 + Dex`; else look up the catalog — Light `base + Dex`, Medium `base + min(Dex, 2)`, Heavy `base`. Shield `+2`. Magic `+ MagicBonus`. **Unarmored Defense** (only when unarmored): compute the Barbarian (`10 + Dex + Con`, shield allowed → may add the +2) and Monk (`10 + Dex + Wis`, ONLY if no shield) options for classes the character has, and take `max(base-unarmored, applicable UD options)`. Total = the chosen base + shield + magic. Components: `armor` (or `unarmored defense (<Class>)`), `dex`, `shield`, `magic`, with the total in `Value`. Computed → null provenance. Unknown armor name → `ResolvedFact("armor class", "unknown armor", [], "needsReview")`.
- **D4 — Unarmored Defense class rule (static).** A small static set/helper: Barbarian → `10 + Dex + Con` (shield allowed); Monk → `10 + Dex + Wis` (no shield). Detected across `sheet.Classes` (a multiclass Barbarian/Monk gets the option). If both apply, take the higher. This is the same static-class-data discipline as `SavingThrowProficiencies` — no corpus dependency.
- **D5 — UI (capture only).** In `HeroDetail.razor`'s edit form, near the AC input: a `<select>` bound to `_editSheet.WornArmor.ArmorName` (options: `None` + the catalog names, sourced from `ArmorCatalog`), a shield `<input type="checkbox">` bound to `WornArmor.Shield`, and a magic-bonus `<input type="number">` bound to `WornArmor.MagicBonus`. Ensure `WornArmor` is non-null when editing (initialize on load, like other sheet sub-objects). No AC computation in the UI — it only captures data; the resolver computes AC on demand. This is a presentational + data-capture change: existing behavior/handlers unchanged.

## Risks / Trade-offs

- **Per-branch correctness (D3/D4).** AC branches on armor category + Unarmored Defense class; a single-fixture smoke walks one branch. MITIGATION: a unit test PER branch — unarmored+Barbarian UD, unarmored+Monk UD (and Monk-with-shield → UD suppressed), Light (full Dex), Medium (Dex cap 2), Heavy (no Dex), shield +2, magic bonus, unknown-armor → `needsReview`, multiclass-UD-picks-higher.
- **Unarmored Defense edge cases.** Monk UD requires no shield; Barbarian UD allows a shield. A wrong shield rule silently mis-computes. MITIGATION: explicit Monk-with-shield test (UD suppressed → falls back to `10 + Dex` + shield) and Barbarian-with-shield test (UD + shield).
- **Old-snapshot deserialization.** A pre-change snapshot has no `WornArmor` key → must deserialize to a non-null default (unarmored), and the resolver must treat that as unarmored (not throw). MITIGATION: default-initialize `WornArmor` (property initializer) and a unit test resolving a sheet with a default/empty `WornArmor` → unarmored AC.
- **UI-can't-be-unit-tested.** The dropdown/checkbox render is invisible to build/test. MITIGATION: per the presentational gates — full suite stays green (behavior unchanged), every referenced CSS class resolves, and a Playwright screenshot of the armor controls (desktop + mobile) after a container rebuild.
- **Manual `ArmorClass` coexistence.** The sheet keeps a separate manual AC int; the resolver's computed AC may differ. Accepted — they are distinct (one is the user's recorded value, the other a derived fact); the resolver never overwrites the stored value.
