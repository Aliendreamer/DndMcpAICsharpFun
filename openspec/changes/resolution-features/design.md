## Context

`CharacterResolutionService.ResolveForSheetAsync` dispatches a feature string to a `Resolve…Async` method; each returns a `ResolvedFact(Feature, Value, Components, Confidence)` where every `ResolvedComponent` carries an optional `ProvenanceRef`. Table-backed features read `StructuredTables`/`StructuredTableRows` by `CanonicalId` (class tables via `.EndsWith($".table.{Slug(class)}")`, exactly as `ResolveClassFeaturesAsync` does). Computed features (spell save DC/attack) carry null provenance. The grounding contract is: value when grounded, `"needsReview"` when a required cell/table is absent, and **never a fabricated default**.

Verified grounding (direct scan of `books/canonical/phb14.json` `tables[]`): the projected class progression tables carry the columns these features read — e.g. barbarian `[Level, Proficiency Bonus, Features, Rages, Rage Damage]`, monk `[…, Martial Arts, Ki Points, Unarmored Movement]`, sorcerer `[…, Sorcery Points, Cantrips Known, Spells Known, 1st…9th]`, rogue `[…, Sneak Attack]`, warlock `[…, Cantrips Known, Spells Known, Spell Slots, Slot Level, Invocations Known]`, bard/ranger `[…, Spells Known, …]`. Prepared casters (cleric/druid/wizard/paladin) have no `Spells Known` column — they compute.

## Goals / Non-Goals

**Goals:** add `class resources`, `saving throws`, `spell count` as grounded, cited, multiclass-aware resolver features reusing existing data. **Non-Goals:** Armor Class (deferred to `structured-armor-and-ac`); any `CharacterSheet` schema change; new endpoints; re-projecting or editing the class tables; recovering resource data for classes whose column is absent (honest `"none"`/`needsReview` instead).

## Decisions

- **D1 — `class resources` column selection.** Emit every class-table column EXCEPT the structural/spellcasting set `{Level, Proficiency Bonus, Features, Cantrips Known, Spells Known, Spell Slots, Slot Level, "1st".."9th"}` (case-insensitive). The remainder are the class-specific resources. Read the row at `RowIndex == c.Level - 1`; each non-empty resource cell → a component labeled `"<Class> <Column>"` with the cell value + provenance. No resource columns for the class → `"none"` (honest, `ok`). Missing table → `needsReview`. Ambiguous (>1 book defines the suffix) → `needsReview`, mirroring `ResolveClassFeaturesAsync`.
- **D2 — `saving throws` proficiency source.** A static `class → (Ability, Ability)` map (PHB, all 12 classes; same shape/spirit as `MulticlassSpellcasting.SpellcastingAbility`). Proficient saves come from the STARTING class only (`sheet.Classes[0]`) — 5e grants no saving-throw proficiency from multiclassing. Emit all six abilities: `value = Modifier(score) + (proficient ? ProficiencyBonus : 0)`, component label = the ability name, `proficient` reflected in the value string (e.g. `"+5 (proficient)"` / `"+1"`). Computed → null provenance. Starting class not in the map → `needsReview` for the whole fact.
- **D3 — `spell count` caster-type branching (DATA-DRIVEN, not slot-source).** The known-vs-prepared axis is NOT the slot-source axis (full/half/third/pact) — Warlock is pact for slots but has a `Spells Known` column. So classify by the DATA: (a) KNOWN caster ⇔ the class table row has a `Spells Known` cell → read it (table-grounded, cell provenance) [Bard/Sorcerer/Ranger/Warlock]; (b) PREPARED caster ⇔ no `Spells Known` column but `SpellcastingAbility(class) != null` → compute via `SpellcastingAbility`, formula by caster tier: full-caster `mod + level`, half-caster (`MulticlassSpellcasting.Classify(c)` is half — Paladin) `mod + level/2`, both min 1 (computed, null provenance) [Cleric/Druid/Wizard full; Paladin half]; (c) NON-caster ⇔ `SpellcastingAbility(class) == null` → contributes `"no spellcasting"` (no fabricated number). `Cantrips Known` cell added where that column exists. Because classification is DATA-DRIVEN (known ⇔ a `Spells Known` cell exists), a known caster whose table is absent cannot be distinguished from a prepared caster, so a class with no `Spells Known` cell but a non-null spellcasting ability computes the prepared formula — accepted, since the corpus carries every class table (this is the one edge the data-driven scheme trades away for simplicity). If no class contributes any count → `needsReview`.
- **D4 — reuse, don't duplicate.** Reuse `ResolveClassFeaturesAsync`'s table-lookup shape (suffix match + Take(2) ambiguity guard), `MulticlassSpellcasting.SpellcastingAbility`/caster-type, `CharacterSheet.Modifier`/`ProficiencyBonus`, `EntityIdSlug.Slug`. The only new static data is the class→saves map.

## Risks / Trade-offs

- **Per-branch correctness (D3).** `spell count` branches on caster type; a single-fixture smoke walks one branch. MITIGATION: a unit test PER branch (known / prepared-full / prepared-half / non-caster / warlock), asserting the excluded/formula branches too — the recurring "test each classifier value" discipline.
- **Column-name drift.** D1/D3 match column names from the 5etools projection (`"Spells Known"`, `"Sorcery Points"`). If a future projection renames a column, the feature silently reads nothing. MITIGATION: match case-insensitively; the integration test seeds the real column names and asserts a grounded value, so a rename breaks a test rather than shipping silent-empty.
- **Multiclass save semantics (D2).** Using only `Classes[0]` for save proficiency is the correct 5e rule but assumes list order = "starting class". Accepted: the sheet treats `Classes[0]` as primary elsewhere (`CharacterSheet.Class`).
- **Real-infra grounding.** Unit fakes prove parse/branch logic, not that the real corpus tables carry the columns. MITIGATION: one real-Postgres integration test (reuse `PostgresFixture`) seeding a class table with the real column names and asserting resolved value + `Confidence=="ok"` — plus grounding was pre-verified against `phb14.json`.
